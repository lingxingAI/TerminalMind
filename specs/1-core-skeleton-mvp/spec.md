# 核心骨架 MVP — 功能规约

> 分支：1-core-skeleton-mvp
> 创建日期：2026-03-25
> 状态：草稿

## 概述

TerminalMind 核心骨架 MVP 是产品的第一个可交付里程碑。目标是构建一个可运行的跨平台（Windows / macOS / Linux）Electron 桌面应用，支持多标签本地终端，并从第一天起建立完整的五层架构和扩展注册机制。本阶段的核心价值在于验证架构设计的可行性——证明内置功能可以通过 Extension API 注册，Core CLI 层可以脱离 GUI 独立运行和测试，以及五层分层架构的数据单向流动模型。

## 用户场景

### 参与者

| 参与者 | 描述 |
|---|---|
| 全栈开发者 | 日常使用终端进行开发和运维的主要用户，需要高效的多标签终端体验 |
| 扩展开发者 | 未来需要基于 Extension API 开发第三方插件的开发者（本阶段作为 API 设计的验证参考） |

### 用户故事

#### US1：多标签本地终端（P1）

**作为**全栈开发者，**我想要**在一个应用窗口中打开多个本地终端标签页，**以便**同时处理不同任务（如运行服务、查看日志、执行命令）而无需在多个窗口间切换。

**验收场景：**

1. **场景**：新建终端标签
   - **假设** 应用已启动且主界面已加载
   - **当** 用户点击新建终端按钮或使用快捷键
   - **则** 在标签栏新增一个终端标签，主工作区显示新终端会话，光标处于就绪状态

2. **场景**：切换终端标签
   - **假设** 用户已打开多个终端标签
   - **当** 用户点击另一个标签
   - **则** 主工作区切换到对应终端会话，之前的会话状态保持不变

3. **场景**：关闭终端标签
   - **假设** 用户有正在运行的终端会话
   - **当** 用户点击标签的关闭按钮
   - **则** 对应终端会话被终止，标签从标签栏移除，焦点切换到相邻标签

#### US2：Shell 选择（P1）

**作为**全栈开发者，**我想要**选择使用哪种 Shell（如 PowerShell、Bash、Zsh）来创建新终端，**以便**根据不同任务需求使用最合适的 Shell 环境。

**验收场景：**

1. **场景**：使用默认 Shell 创建终端
   - **假设** 用户未手动选择 Shell 类型
   - **当** 用户新建终端
   - **则** 使用系统默认 Shell 或用户配置的默认 Shell 创建终端会话

2. **场景**：指定 Shell 创建终端
   - **假设** 用户通过命令面板或 UI 选择了特定 Shell
   - **当** 终端创建完成
   - **则** 新终端使用用户指定的 Shell 运行

#### US3：命令面板快速操作（P2）

**作为**全栈开发者，**我想要**通过快捷键唤出命令面板，输入关键词搜索和执行命令，**以便**快速访问各种功能而无需记忆菜单位置。

**验收场景：**

1. **场景**：唤出命令面板
   - **假设** 应用处于任意视图
   - **当** 用户按下 `Ctrl+Shift+P`（macOS 为 `Cmd+Shift+P`）
   - **则** 屏幕中央上方弹出命令面板输入框，列出最近使用的命令

2. **场景**：搜索并执行命令
   - **假设** 命令面板已打开
   - **当** 用户输入关键词（如 "terminal"）
   - **则** 实时过滤显示匹配的命令列表，用户选择后执行对应命令

#### US4：VS Code 风格布局（P2）

**作为**全栈开发者，**我想要**应用采用类似 VS Code 的布局（活动栏、侧边栏、主工作区、底部面板），**以便**利用已有的 VS Code 使用经验，减少学习成本。

**验收场景：**

1. **场景**：识别主要布局区域
   - **假设** 应用已启动
   - **当** 用户查看应用界面
   - **则** 可清晰辨别活动栏（左侧图标）、侧边栏（左面板）、标签栏+主工作区（中央）、底部面板区域和状态栏

2. **场景**：通过活动栏切换侧边栏视图
   - **假设** 侧边栏当前显示某一视图
   - **当** 用户点击活动栏上的另一个图标
   - **则** 侧边栏切换到对应视图内容

#### US5：终端交互体验（P1）

**作为**全栈开发者，**我想要**终端支持标准的交互功能（文本输入输出、滚动回看、窗口大小自适应），**以便**获得与原生终端一致的使用体验。

**验收场景：**

1. **场景**：执行命令并查看输出
   - **假设** 终端已打开并处于就绪状态
   - **当** 用户输入命令并按下回车
   - **则** 命令被执行，输出在终端中正确显示，支持 ANSI 颜色和格式

2. **场景**：滚动回看历史输出
   - **假设** 终端中有大量历史输出超出可视区域
   - **当** 用户向上滚动
   - **则** 可以查看之前的命令和输出内容

3. **场景**：窗口大小变更
   - **假设** 终端正在运行
   - **当** 用户调整应用窗口大小或面板尺寸
   - **则** 终端自动适应新的行列数，重新渲染内容

#### US6：CLI 模式独立运行（P1）

**作为**扩展开发者，**我想要** Core CLI 层可以在无 GUI 的 Node.js 环境中独立运行，**以便**编写和执行单元测试时无需启动 Electron。

**验收场景：**

1. **场景**：CLI 模式注册和执行命令
   - **假设** 在纯 Node.js 环境（无 Electron）
   - **当** 通过 Core CLI 注册一个命令并执行
   - **则** 命令正确执行并返回结果，无 Electron 依赖报错

2. **场景**：扩展通过 API 注册
   - **假设** 在纯 Node.js 测试环境
   - **当** ext-terminal 扩展通过 Extension API 的 activate 生命周期注册命令
   - **则** 命令成功注册到 CommandRegistry，可通过 CLI 方式调用

## 成功标准

| 标准 | 指标 | 目标值 |
|---|---|---|
| 应用启动速度 | 从双击图标到终端可交互的时间 | < 3 秒 |
| 终端响应性 | 用户按键到字符在终端显示的延迟 | < 50 毫秒（用户无感知延迟） |
| 架构可验证性 | Core CLI 层可脱离 GUI 运行单元测试 | 100% core 和 services 测试无需 Electron |
| 扩展 API 一致性 | 内置扩展使用与第三方相同的 API | ext-terminal 无任何特权 API 调用 |
| 多标签稳定性 | 同时打开多个终端标签时应用的稳定性 | 同时运行 10 个终端标签无崩溃、无内存泄漏 |
| 跨平台兼容性 | 应用在三大操作系统上的可运行性 | Windows、macOS、Linux 均可正常构建和运行 |
| 终端兼容性 | 终端对 ANSI 转义序列的支持度 | 正确渲染颜色、粗体、下划线等常见格式 |
| 测试覆盖率 | Core CLI 和 Services 层的单元测试覆盖 | 每个 Service 和 Command 至少有一个正常路径测试 |

## 边界情况

- **Shell 不可用**：用户系统中配置的默认 Shell 已卸载或路径失效时，应用应回退到系统最基础的 Shell（Windows 上为 cmd.exe，macOS/Linux 上为 /bin/sh），并向用户显示提示信息
- **终端进程异常退出**：当 PTY 进程被外部信号终止或崩溃时，终端标签应显示退出状态码，标签不自动关闭，由用户决定关闭或重新打开
- **大量输出冲刷**：当命令产生海量输出（如 `cat` 大文件）时，终端渲染不应导致 UI 线程阻塞，滚动回看缓冲区应有上限（默认 10000 行）
- **窗口极小化**：当用户将应用窗口调整到极小尺寸时，布局应优雅降级，不出现元素重叠或溢出
- **多平台路径差异**：Windows 使用反斜杠路径和驱动器号，macOS/Linux 使用正斜杠，Shell 发现和 PTY 创建需正确处理平台差异
- **快速创建/关闭标签**：用户快速连续创建或关闭大量终端标签时，应用不应出现竞态条件或资源泄漏
- **长时间运行进程**：终端中运行长时间进程（如 `top`、`watch`）时，应用不应因持续输出而产生内存泄漏
- **IPC 通信中断**：Main 进程和 Renderer 进程间的 IPC 通道异常时，应优雅处理而非崩溃

## 依赖

- **node-pty**：跨平台伪终端实现，用于创建本地终端会话
- **xterm.js + xterm-addon-fit + xterm-addon-webgl**：终端渲染库及自适应尺寸插件
- **Electron**：跨平台桌面应用框架
- **React 18+**：GUI Shell 前端框架
- **Zustand**：UI 状态管理
- **Vite + electron-builder**：构建工具链
- **pnpm workspaces**：Monorepo 包管理
- **Vitest**：单元测试框架
- **ESLint + Prettier**：代码规范工具
- **TypeScript (strict)**：编程语言

## 假设

- 用户的操作系统已安装至少一种 Shell（Windows 自带 PowerShell 和 cmd，macOS/Linux 自带 bash 或 zsh）
- 开发和构建环境已安装 Node.js 18+ 和 pnpm 8+
- 首个版本不需要支持无障碍访问（WCAG），但布局设计应预留后续改造空间
- 底部面板在本阶段仅需渲染占位布局，无需实现 AI Chat 或 Output 面板的具体功能
- 活动栏图标（连接、文件、搜索、AI、扩展、设置）在本阶段全部显示，但仅"终端"相关视图有实际内容，其余点击后显示空白占位或"即将推出"提示
- ConfigService 在本阶段仅需支持基础的 Shell 默认配置读写，完整的分层配置系统推迟到后续阶段
- 性能基准（启动 < 3 秒）以中等配置开发机（8GB RAM、SSD）为测试标准
