## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果非空）。

## 目标

在实现之前，识别三个核心产物（`spec.md`、`plan.md`、`tasks.md`）之间的不一致、重复、歧义和不明确项。此命令必须在 `/speckit.tasks` 成功生成完整的 `tasks.md` 后才能运行。

## 运行约束

**严格只读**：**不要**修改任何文件。输出结构化分析报告。提供可选的修复计划（用户必须明确批准后才能手动调用后续编辑命令）。

**章程权威**：项目章程（`.specify/memory/constitution.md`）在本分析范围内**不可协商**。章程冲突自动为 CRITICAL，需要调整规约、计划或任务——而非稀释、重新解释或静默忽略原则。如果原则本身需要变更，必须在 `/speckit.analyze` 之外通过单独的、明确的章程更新进行。

## 语言规则

分析报告中所有自然语言内容必须使用简体中文编写。包括报告标题、发现摘要、建议、后续操作和修复建议。技术标识符（发现 ID 如 A1/B2、严重性标签、文件路径、行引用、需求键）保持英文。

## 执行步骤

### 1. 初始化分析上下文

从仓库根目录运行 `.specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks` 一次，解析 JSON 获取 FEATURE_DIR 和 AVAILABLE_DOCS。推导绝对路径：

- SPEC = FEATURE_DIR/spec.md
- PLAN = FEATURE_DIR/plan.md
- TASKS = FEATURE_DIR/tasks.md

如果任何必需文件缺失，中止并显示错误消息（指示用户运行缺失的前置命令）。
对于参数中的单引号（如 "I'm Groot"），使用转义语法：如 'I'\''m Groot'（或尽可能使用双引号："I'm Groot"）。

### 2. 加载产物（渐进式披露）

仅从每个产物加载最小必要上下文：

**来自 spec.md：**

- 概述/上下文
- 功能需求
- 非功能需求
- 用户故事
- 边界情况（如存在）

**来自 plan.md：**

- 架构/技术栈选择
- 数据模型引用
- 阶段
- 技术约束

**来自 tasks.md：**

- 任务 ID
- 描述
- 阶段分组
- 并行标记 [P]
- 引用的文件路径

**来自章程：**

- 加载 `.specify/memory/constitution.md` 用于原则验证

### 3. 构建语义模型

创建内部表示（不在输出中包含原始产物）：

- **需求清单**：每个功能 + 非功能需求，带稳定键（基于祈使短语推导 slug；如"用户可上传文件" → `user-can-upload-file`）
- **用户故事/操作清单**：离散用户操作及验收标准
- **任务覆盖映射**：将每个任务映射到一个或多个需求或故事（通过关键词推断/明确引用模式如 ID 或关键短语）
- **章程规则集**：提取原则名称和 MUST/SHOULD 规范性声明

### 4. 检测遍历（Token 高效分析）

聚焦于高信号发现。限制总计 50 个发现；其余汇总在溢出摘要中。

#### A. 重复检测

- 识别近似重复的需求
- 标记质量较低的措辞以便合并

#### B. 歧义检测

- 标记缺乏可衡量标准的模糊形容词（快速、可扩展、安全、直观、健壮）
- 标记未解决的占位符（TODO、TKTK、???、`<placeholder>` 等）

#### C. 不明确

- 有动词但缺少对象或可衡量结果的需求
- 缺少验收标准对齐的用户故事
- 引用了规约/计划中未定义的文件或组件的任务

#### D. 章程对齐

- 任何与 MUST 原则冲突的需求或计划元素
- 章程中要求的章节或质量关卡缺失

#### E. 覆盖缺口

- 零关联任务的需求
- 无映射需求/故事的任务
- 未在任务中反映的非功能需求（如性能、安全）

#### F. 不一致

- 术语漂移（同一概念在不同文件中命名不同）
- 计划中引用但规约中缺失的数据实体（或反之）
- 任务排序矛盾（如集成任务在基础初始化任务之前且无依赖说明）
- 冲突的需求（如一个要求 Next.js 而另一个指定 Vue）

### 5. 严重性分配

使用以下启发式进行优先级排序：

- **CRITICAL**：违反章程 MUST、缺少核心规约产物、或零覆盖的需求阻塞基线功能
- **HIGH**：重复或冲突的需求、模糊的安全/性能属性、不可测试的验收标准
- **MEDIUM**：术语漂移、缺失的非功能任务覆盖、不明确的边界情况
- **LOW**：风格/措辞改进、不影响执行顺序的轻微冗余

### 6. 生成紧凑分析报告

输出 Markdown 报告（不写入文件），结构如下：

## 规约分析报告

| ID | 类别 | 严重性 | 位置 | 摘要 | 建议 |
|----|------|--------|------|------|------|
| A1 | 重复 | HIGH | spec.md:L120-134 | 两个相似的需求... | 合并措辞；保留更清晰的版本 |

（每个发现一行；生成以类别首字母为前缀的稳定 ID。）

**覆盖度总结表：**

| 需求键 | 有任务？ | 任务 ID | 备注 |
|--------|---------|---------|------|

**章程对齐问题：**（如有）

**未映射任务：**（如有）

**指标：**

- 总需求数
- 总任务数
- 覆盖率 %（有 >=1 个任务的需求）
- 歧义数
- 重复数
- 关键问题数

### 7. 提供后续操作

在报告末尾输出简洁的后续操作块：

- 如果存在 CRITICAL 问题：建议在 `/speckit.implement` 前解决
- 如果仅有 LOW/MEDIUM：用户可继续，但提供改进建议
- 提供明确的命令建议：如"运行 /speckit.specify 进行细化"、"运行 /speckit.plan 调整架构"、"手动编辑 tasks.md 为 'performance-metrics' 添加覆盖"

### 8. 提供修复方案

询问用户："是否需要我为排名前 N 的问题建议具体的修复编辑？"（不要自动应用。）

## 运行原则

### 上下文效率

- **最小高信号 token**：聚焦于可操作的发现，而非详尽的文档
- **渐进式披露**：增量加载产物；不要将所有内容倾倒到分析中
- **Token 高效输出**：发现表限制 50 行；总结溢出部分
- **确定性结果**：无变更时重新运行应产生一致的 ID 和计数

### 分析指南

- **绝不修改文件**（这是只读分析）
- **绝不臆造缺失章节**（如果缺失，准确报告）
- **优先处理章程违规**（这些始终为 CRITICAL）
- **用示例而非详尽规则**（引用具体实例，而非通用模式）
- **优雅地报告零问题**（输出带覆盖统计的成功报告）
