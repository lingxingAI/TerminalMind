## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果非空）。

## 执行大纲

用户在触发消息中 `/speckit.specify` 后面输入的文本**就是**功能描述。即使下面 `$ARGUMENTS` 字面出现，也假设你始终可以在本次对话中获取到它。除非用户提供了空命令，否则不要要求用户重复输入。

**语言规则**: 生成的规约文档中所有自然语言内容必须使用简体中文编写。包括章节标题、用户故事、需求、验收场景、成功标准、边界情况、澄清问题及所有描述性文本。技术标识符（FR-001、SC-001、[NEEDS CLARIFICATION]、文件路径、代码片段）保持英文。

根据功能描述，执行以下步骤：

1. **生成简洁的短名称**（2-4 个词）用于分支命名：
   - 分析功能描述，提取最有意义的关键词
   - 创建 2-4 个词的短名称，捕捉功能的核心
   - 尽可能使用"动作-名词"格式（如 "add-user-auth"、"fix-payment-bug"）
   - 保留技术术语和缩写（OAuth2、API、JWT 等）
   - 保持简洁但足够描述性，一眼即可理解功能
   - 示例：
     - "我想添加用户认证" → "user-auth"
     - "为 API 实现 OAuth2 集成" → "oauth2-api-integration"
     - "创建分析仪表盘" → "analytics-dashboard"
     - "修复支付处理超时 bug" → "fix-payment-timeout"

2. **创建新分支前检查已有分支**：

   a. 首先拉取所有远程分支以确保信息最新：

      ```bash
      git fetch --all --prune
      ```

   b. 在所有来源中查找该 short-name 的最大功能编号：
      - 远程分支：`git ls-remote --heads origin | grep -E 'refs/heads/[0-9]+-<short-name>$'`
      - 本地分支：`git branch | grep -E '^[* ]*[0-9]+-<short-name>$'`
      - Specs 目录：检查匹配 `specs/[0-9]+-<short-name>` 的目录

   c. 确定下一个可用编号：
      - 从所有三个来源提取所有编号
      - 找到最大编号 N
      - 使用 N+1 作为新分支编号

   d. 运行脚本 `.specify/scripts/bash/create-new-feature.sh --json "$ARGUMENTS"`，传入计算出的编号和 short-name：
      - 传递 `--number N+1` 和 `--short-name "your-short-name"` 以及功能描述
      - Bash 示例：`.specify/scripts/bash/create-new-feature.sh --json "$ARGUMENTS" --json --number 5 --short-name "user-auth" "Add user authentication"`
      - PowerShell 示例：`.specify/scripts/bash/create-new-feature.sh --json "$ARGUMENTS" -Json -Number 5 -ShortName "user-auth" "Add user authentication"`

   **重要**：
   - 检查所有三个来源（远程分支、本地分支、specs 目录）以找到最大编号
   - 仅匹配精确 short-name 模式的分支/目录
   - 如果未找到该 short-name 的已有分支/目录，从编号 1 开始
   - 每个功能只能运行此脚本一次
   - JSON 作为终端输出提供——始终引用它来获取你需要的实际内容
   - JSON 输出将包含 BRANCH_NAME 和 SPEC_FILE 路径
   - 对于参数中的单引号（如 "I'm Groot"），使用转义语法：如 'I'\''m Groot'（或尽可能使用双引号："I'm Groot"）

3. 加载 `.specify/templates/spec-template.md` 以了解必需的章节。

4. 按以下执行流程操作：

    1. 从输入中解析用户描述
       如果为空：ERROR "未提供功能描述"
    2. 从描述中提取关键概念
       识别：参与者、操作、数据、约束
    3. 对于不明确的方面：
       - 基于上下文和行业标准做出合理推断
       - 仅在以下情况标记 [NEEDS CLARIFICATION: 具体问题]：
         - 该选择显著影响功能范围或用户体验
         - 存在多种合理解释且含义不同
         - 不存在合理的默认值
       - **限制：最多 3 个 [NEEDS CLARIFICATION] 标记**
       - 按影响优先级排列澄清项：范围 > 安全/隐私 > 用户体验 > 技术细节
    4. 填写用户场景与测试章节
       如果无法确定用户流程：ERROR "无法确定用户场景"
    5. 生成功能需求
       每个需求必须可测试
       对未指定的细节使用合理默认值（在假设章节中记录假设）
    6. 定义成功标准
       创建可衡量的、与技术无关的成果
       包含定量指标（时间、性能、数量）和定性指标（用户满意度、任务完成率）
       每个标准必须无需实现细节即可验证
    7. 识别关键实体（如涉及数据）
    8. 返回：SUCCESS（规约已准备好进入计划阶段）

5. 使用模板结构将规约写入 SPEC_FILE，用从功能描述（参数）推导出的具体细节替换占位符，同时保持章节顺序和标题。

6. **规约质量验证**：写入初始规约后，根据质量标准进行验证：

   a. **创建规约质量检查清单**：在 `FEATURE_DIR/checklists/requirements.md` 生成检查清单文件，使用检查清单模板结构，包含以下验证项：

      ```markdown
      # 规约质量检查清单: [FEATURE NAME]
      
      **用途**: 在进入计划阶段前验证规约的完整性和质量
      **创建日期**: [DATE]
      **功能**: [Link to spec.md]
      
      ## 内容质量
      
      - [ ] 不包含实现细节（语言、框架、API）
      - [ ] 聚焦于用户价值和业务需求
      - [ ] 面向非技术干系人编写
      - [ ] 所有必填章节已完成
      
      ## 需求完整性
      
      - [ ] 无残留的 [NEEDS CLARIFICATION] 标记
      - [ ] 需求可测试且无歧义
      - [ ] 成功标准可衡量
      - [ ] 成功标准与技术无关（不含实现细节）
      - [ ] 所有验收场景已定义
      - [ ] 边界情况已识别
      - [ ] 范围边界清晰
      - [ ] 依赖和假设已识别
      
      ## 功能就绪度
      
      - [ ] 所有功能需求有明确的验收标准
      - [ ] 用户场景覆盖主要流程
      - [ ] 功能满足成功标准中定义的可衡量成果
      - [ ] 规约中未泄露实现细节
      
      ## 注意事项
      
      - 标记为未完成的项目需要在执行 `/speckit.clarify` 或 `/speckit.plan` 前更新规约
      ```

   b. **执行验证检查**：根据每个检查项审查规约：
      - 对每个项目判断通过或失败
      - 记录发现的具体问题（引用相关规约章节）

   c. **处理验证结果**：

      - **如果所有项目通过**：标记检查清单完成，继续步骤 6

      - **如果项目失败（不含 [NEEDS CLARIFICATION]）**：
        1. 列出失败项目和具体问题
        2. 更新规约以解决每个问题
        3. 重新运行验证直到所有项目通过（最多 3 次迭代）
        4. 如果 3 次迭代后仍有失败，在检查清单注意事项中记录剩余问题并警告用户

      - **如果残留 [NEEDS CLARIFICATION] 标记**：
        1. 从规约中提取所有 [NEEDS CLARIFICATION: ...] 标记
        2. **限制检查**：如果超过 3 个标记，仅保留最关键的 3 个（按范围/安全/UX 影响排序），其余做合理推断
        3. 对每个需要澄清的项目（最多 3 个），以以下格式向用户展示选项：

           ```markdown
           ## 问题 [N]: [主题]
           
           **上下文**: [引用相关规约章节]
           
           **需要了解的内容**: [NEEDS CLARIFICATION 标记中的具体问题]
           
           **建议答案**:
           
           | 选项 | 答案 | 影响 |
           |------|------|------|
           | A    | [第一个建议答案] | [这对功能意味着什么] |
           | B    | [第二个建议答案] | [这对功能意味着什么] |
           | C    | [第三个建议答案] | [这对功能意味着什么] |
           | 自定义 | 提供你自己的答案 | [说明如何提供自定义输入] |
           
           **你的选择**: _[等待用户回复]_
           ```

        4. **关键 - 表格格式**：确保 Markdown 表格格式正确：
           - 使用一致的间距，管道符对齐
           - 每个单元格内容周围有空格：`| 内容 |` 而非 `|内容|`
           - 表头分隔符至少 3 个短横线：`|--------|`
           - 测试表格在 Markdown 预览中正确渲染
        5. 问题按顺序编号（Q1、Q2、Q3 - 最多 3 个）
        6. 在等待回复前一次性展示所有问题
        7. 等待用户对所有问题做出选择（如 "Q1: A, Q2: 自定义 - [详情], Q3: B"）
        8. 用用户选择或提供的答案替换规约中的每个 [NEEDS CLARIFICATION] 标记
        9. 所有澄清解决后重新运行验证

   d. **更新检查清单**：每次验证迭代后，更新检查清单文件的通过/失败状态

7. 报告完成情况，包括分支名称、规约文件路径、检查清单结果，以及进入下一阶段（`/speckit.clarify` 或 `/speckit.plan`）的就绪状态。

**注意**：脚本会在写入前创建并切换到新分支，并初始化规约文件。

## 通用指南

## 快速指南

- 聚焦于用户**需要什么**以及**为什么**需要。
- 避免涉及如何实现（不涉及技术栈、API、代码结构）。
- 面向业务干系人编写，而非开发者。
- 不要在规约中嵌入检查清单。那将由单独的命令处理。

### 章节要求

- **必填章节**：每个功能都必须完成
- **可选章节**：仅在与功能相关时包含
- 当某章节不适用时，完全删除它（不要留 "N/A"）

### AI 生成指南

从用户提示创建规约时：

1. **做出合理推断**：利用上下文、行业标准和常见模式填补空白
2. **记录假设**：在假设章节中记录合理的默认值
3. **限制澄清项**：最多 3 个 [NEEDS CLARIFICATION] 标记——仅用于以下关键决策：
   - 显著影响功能范围或用户体验
   - 存在多种合理解释且含义不同
   - 不存在合理的默认值
4. **澄清优先级**：范围 > 安全/隐私 > 用户体验 > 技术细节
5. **像测试人员一样思考**：每个模糊的需求都应该无法通过"可测试且无歧义"的检查项
6. **常见需要澄清的领域**（仅在不存在合理默认值时）：
   - 功能范围和边界（包含/排除特定用例）
   - 用户类型和权限（如果存在多种冲突解释）
   - 安全/合规要求（当涉及法律/财务重大影响时）

**合理默认值示例**（不要询问这些）：

- 数据保留：该领域的行业标准做法
- 性能目标：标准 Web/移动应用期望值（除非另有指定）
- 错误处理：用户友好的消息和适当的降级方案
- 认证方式：Web 应用的标准会话或 OAuth2
- 集成模式：使用项目适配的模式（Web 服务用 REST/GraphQL，库用函数调用，工具用 CLI 参数等）

### 成功标准指南

成功标准必须：

1. **可衡量**：包含具体指标（时间、百分比、数量、比率）
2. **与技术无关**：不提及框架、语言、数据库或工具
3. **以用户为中心**：从用户/业务角度描述成果，而非系统内部
4. **可验证**：无需了解实现细节即可测试/验证

**好的示例**：

- "用户可在 3 分钟内完成结账"
- "系统支持 10,000 个并发用户"
- "95% 的搜索在 1 秒内返回结果"
- "任务完成率提高 40%"

**不好的示例**（聚焦实现）：

- "API 响应时间低于 200ms"（太技术化，应使用"用户即时看到结果"）
- "数据库可处理 1000 TPS"（实现细节，应使用面向用户的指标）
- "React 组件高效渲染"（框架特定）
- "Redis 缓存命中率超过 80%"（技术特定）
