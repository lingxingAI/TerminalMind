## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果非空）。

## 执行大纲

**目标**：检测并减少当前功能规约中的歧义或缺失的决策点，并将澄清内容直接记录到规约文件中。

**语言规则**: 所有自然语言输出（澄清问题、推荐、覆盖度总结、完成报告）以及写回规约文件的所有内容（澄清记录、更新的需求、边界情况）必须使用简体中文。技术标识符（章节标记、文件路径、代码片段）保持英文。

**注意**：此澄清工作流预期在调用 `/speckit.plan` 之前运行（并完成）。如果用户明确表示跳过澄清（如探索性 spike），你可以继续，但必须警告下游返工风险增加。

执行步骤：

1. 从仓库根目录运行 `.specify/scripts/bash/check-prerequisites.sh --json --paths-only` **一次**（组合 `--json --paths-only` 模式 / `-Json -PathsOnly`）。解析最小 JSON 负载字段：
   - `FEATURE_DIR`
   - `FEATURE_SPEC`
   - （可选捕获 `IMPL_PLAN`、`TASKS` 用于后续链式流程。）
   - 如果 JSON 解析失败，中止并指示用户重新运行 `/speckit.specify` 或验证功能分支环境。
   - 对于参数中的单引号（如 "I'm Groot"），使用转义语法：如 'I'\''m Groot'（或尽可能使用双引号："I'm Groot"）。

2. 加载当前规约文件。使用以下分类法进行结构化歧义和覆盖度扫描。对每个类别标记状态：清晰 / 部分 / 缺失。生成用于优先级排序的内部覆盖度地图（除非不会提问，否则不输出原始地图）。

   功能范围与行为：
   - 核心用户目标和成功标准
   - 明确的范围外声明
   - 用户角色/人物画像区分

   领域与数据模型：
   - 实体、属性、关系
   - 身份和唯一性规则
   - 生命周期/状态转换
   - 数据量/规模假设

   交互与 UX 流程：
   - 关键用户旅程/序列
   - 错误/空/加载状态
   - 无障碍或本地化说明

   非功能质量属性：
   - 性能（延迟、吞吐量目标）
   - 可扩展性（水平/垂直、限制）
   - 可靠性与可用性（正常运行时间、恢复预期）
   - 可观测性（日志、指标、追踪信号）
   - 安全与隐私（认证/授权、数据保护、威胁假设）
   - 合规/监管约束（如有）

   集成与外部依赖：
   - 外部服务/API 及失败模式
   - 数据导入/导出格式
   - 协议/版本假设

   边界情况与故障处理：
   - 负面场景
   - 限流/节流
   - 冲突解决（如并发编辑）

   约束与权衡：
   - 技术约束（语言、存储、托管）
   - 明确的权衡或被拒绝的替代方案

   术语与一致性：
   - 规范术语表
   - 避免的同义词/弃用的术语

   完成信号：
   - 验收标准可测试性
   - 可衡量的完成定义风格指标

   杂项/占位符：
   - TODO 标记/未解决的决策
   - 缺乏量化的模糊形容词（"健壮"、"直观"）

   对于状态为"部分"或"缺失"的每个类别，添加候选问题机会，除非：
   - 澄清不会实质性改变实现或验证策略
   - 信息更适合推迟到计划阶段（内部记录）

3. 内部生成优先级排序的候选澄清问题队列（最多 5 个）。不要一次性全部输出。应用以下约束：
    - 整个会话最多 10 个问题。
    - 每个问题必须可以通过以下方式回答：
       - 简短的多选选择（2-5 个不同的、互斥的选项），或
       - 一个词/短语回答（明确约束："5 个词以内回答"）。
    - 仅包含答案会实质性影响架构、数据建模、任务分解、测试设计、UX 行为、运维就绪或合规验证的问题。
    - 确保类别覆盖平衡：优先覆盖影响最大的未解决类别；避免在单个高影响领域（如安全态势）未解决时提两个低影响问题。
    - 排除已回答的问题、琐碎的风格偏好或计划层面的执行细节（除非阻塞正确性）。
    - 优先选择能减少下游返工风险或防止验收测试偏差的澄清。
    - 如果超过 5 个类别未解决，按（影响 × 不确定性）启发式选择前 5 个。

4. 顺序提问循环（交互式）：
    - 每次只展示一个问题。
    - 对于多选问题：
       - **分析所有选项**并确定**最合适的选项**，基于：
          - 项目类型的最佳实践
          - 类似实现中的常见模式
          - 风险降低（安全、性能、可维护性）
          - 与规约中可见的项目目标或约束的一致性
       - 在顶部突出展示你的**推荐选项**，附清晰的理由（1-2 句解释为何这是最佳选择）。
       - 格式：`**推荐：** 选项 [X] - <理由>`
       - 然后将所有选项渲染为 Markdown 表格：

       | 选项 | 描述 |
       |------|------|
       | A | <选项 A 描述> |
       | B | <选项 B 描述> |
       | C | <选项 C 描述>（根据需要添加 D/E，最多 5 个） |
       | 简答 | 提供不同的简短回答（5 个词以内）（仅在自由回答合适时包含） |

       - 表格后添加：`你可以回复选项字母（如"A"），说"是"或"推荐"接受推荐，或提供你自己的简短回答。`
    - 对于简答风格（无有意义的离散选项）：
       - 基于最佳实践和上下文提供你的**建议答案**。
       - 格式：`**建议：** <你的建议答案> - <简要理由>`
       - 然后输出：`格式：简短回答（5 个词以内）。你可以说"是"或"建议"接受建议，或提供你自己的答案。`
    - 用户回答后：
       - 如果用户回复"是"、"推荐"或"建议"，使用你之前陈述的推荐/建议作为答案。
       - 否则，验证答案是否映射到某个选项或符合 ≤5 个词的约束。
       - 如果模糊，要求快速消歧（仍计入同一问题；不推进）。
       - 满意后，记录到工作记忆中（暂不写入磁盘），移至下一个排队问题。
    - 在以下情况停止提问：
       - 所有关键歧义提前解决（剩余排队项变得不必要），或
       - 用户发出完成信号（"完了"、"好了"、"不用了"），或
       - 达到 5 个已提问题。
    - 不要提前透露未来排队的问题。
    - 如果开始时没有有效问题，立即报告无关键歧义。

5. 每个接受的答案后进行集成（增量更新方式）：
    - 维护规约的内存表示（开始时加载一次）加上原始文件内容。
    - 对于本次会话中第一个集成的答案：
       - 确保存在 `## 澄清` 章节（如缺失，在规约模板中最高层级上下文/概述章节之后创建）。
       - 在其下创建（如不存在）`### 会话 YYYY-MM-DD` 子标题。
    - 接受后立即追加一行：`- 问：<问题> → 答：<最终答案>`。
    - 然后立即将澄清应用到最合适的章节：
       - 功能歧义 → 更新或添加功能需求中的要点。
       - 用户交互/角色区分 → 更新用户故事或角色子章节（如存在），包含澄清的角色、约束或场景。
       - 数据形态/实体 → 更新数据模型（添加字段、类型、关系），保持排序；简洁记录添加的约束。
       - 非功能约束 → 在非功能/质量属性章节添加/修改可衡量标准（将模糊形容词转换为指标或明确目标）。
       - 边界情况/负面流程 → 在边界情况/错误处理下添加新要点（或如果模板提供占位符则创建此子章节）。
       - 术语冲突 → 在整个规约中规范化术语；仅在必要时保留原始术语，添加 `（原称"X"）` 一次。
    - 如果澄清使先前的模糊陈述无效，替换该陈述而非重复；不留过时的矛盾文本。
    - 每次集成后保存规约文件，以最小化上下文丢失风险（原子覆写）。
    - 保持格式：不重新排序无关章节；保持标题层级完整。
    - 每个插入的澄清保持最小化和可测试（避免叙述漂移）。

6. 验证（每次写入后执行 + 最终检查）：
   - 澄清会话中每个接受的答案恰好有一个要点（无重复）。
   - 已提（已接受）问题总数 ≤ 5。
   - 更新的章节中不存在新答案本应解决的模糊占位符。
   - 不存在矛盾的先前陈述（扫描已移除的现已无效的替代选择）。
   - Markdown 结构有效；仅允许新标题：`## 澄清`、`### 会话 YYYY-MM-DD`。
   - 术语一致性：所有更新的章节使用相同的规范术语。

7. 将更新后的规约写回 `FEATURE_SPEC`。

8. 报告完成情况（提问循环结束或提前终止后）：
   - 已提问和已回答的问题数量。
   - 更新后的规约路径。
   - 涉及的章节（列出名称）。
   - 覆盖度总结表，列出每个分类法类别及状态：已解决（原为部分/缺失且已处理）、已推迟（超出问题配额或更适合计划阶段）、清晰（已足够）、待处理（仍为部分/缺失但低影响）。
   - 如有待处理或已推迟项，建议是继续执行 `/speckit.plan` 还是稍后在计划后再运行 `/speckit.clarify`。
   - 建议的下一个命令。

行为规则：

- 如果未发现有意义的歧义（或所有潜在问题都是低影响的），回复："未检测到需要正式澄清的关键歧义。" 并建议继续。
- 如果规约文件缺失，指示用户先运行 `/speckit.specify`（不要在此创建新规约）。
- 不要超过 5 个已提问题（单个问题的澄清重试不计为新问题）。
- 避免推测性的技术栈问题，除非其缺失阻塞了功能清晰度。
- 尊重用户的提前终止信号（"停"、"完了"、"继续"）。
- 如果因完全覆盖而未提问，输出紧凑的覆盖度总结（所有类别为清晰），然后建议推进。
- 如果达到配额但仍有未解决的高影响类别，在"已推迟"下明确标记并附理由。
