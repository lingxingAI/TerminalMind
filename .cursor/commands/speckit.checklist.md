## 检查清单用途："需求的单元测试"

**核心概念**：检查清单是**需求编写的单元测试**——它们验证给定领域中需求的质量、清晰度和完整性。

**不用于验证/测试实现**：

- ❌ 不是"验证按钮点击是否正确"
- ❌ 不是"测试错误处理是否有效"
- ❌ 不是"确认 API 返回 200"
- ❌ 不是检查代码/实现是否匹配规约

**用于需求质量验证**：

- ✅ "是否为所有卡片类型定义了视觉层次需求？"（完整性）
- ✅ "'突出显示'是否用具体的尺寸/定位进行了量化？"（清晰度）
- ✅ "所有交互元素的悬停状态需求是否一致？"（一致性）
- ✅ "是否为键盘导航定义了无障碍需求？"（覆盖度）
- ✅ "规约是否定义了 logo 图片加载失败时的行为？"（边界情况）

**比喻**：如果你的规约是用自然语言编写的代码，那么检查清单就是它的单元测试套件。你在测试需求是否编写良好、完整、无歧义且已准备好实现——而不是测试实现是否有效。

## 用户输入

```text
$ARGUMENTS
```

在继续之前，你**必须**考虑用户输入（如果非空）。

## 执行步骤

**语言规则**: 生成的检查清单中所有自然语言内容必须使用简体中文编写。包括检查清单标题、用途、分类标题、检查项描述、注意事项和完成报告。技术标识符（CHK001、[Spec §X.Y]、[Gap]、[Completeness]、文件路径）保持英文。

1. **初始化**：从仓库根目录运行 `.specify/scripts/bash/check-prerequisites.sh --json`，解析 JSON 获取 FEATURE_DIR 和 AVAILABLE_DOCS 列表。
   - 所有文件路径必须为绝对路径。
   - 对于参数中的单引号（如 "I'm Groot"），使用转义语法：如 'I'\''m Groot'（或尽可能使用双引号："I'm Groot"）。

2. **澄清意图（动态）**：推导最多三个初始上下文澄清问题（无预设目录）。它们必须：
   - 从用户措辞 + 从 spec/plan/tasks 中提取的信号生成
   - 仅询问会实质性改变检查清单内容的信息
   - 如果在 `$ARGUMENTS` 中已经明确，则单独跳过
   - 精确优于广泛

   生成算法：
   1. 提取信号：功能领域关键词（如 auth、latency、UX、API）、风险指标（"critical"、"must"、"compliance"）、干系人提示（"QA"、"review"、"security team"）和明确的交付物（"a11y"、"rollback"、"contracts"）。
   2. 将信号聚类为候选关注领域（最多 4 个），按相关性排序。
   3. 识别可能的受众和时机（作者、审查者、QA、发布），如果未明确。
   4. 检测缺失维度：范围广度、深度/严格度、风险重点、排除边界、可衡量的验收标准。
   5. 从以下原型中选择问题：
      - 范围细化（如"是否应包含与 X 和 Y 的集成触点，还是限于本地模块正确性？"）
      - 风险优先级（如"以下哪些潜在风险领域应接受强制关卡检查？"）
      - 深度校准（如"这是轻量级的提交前检查清单还是正式的发布关卡？"）
      - 受众定位（如"这仅由作者使用还是在 PR 审查中由同事使用？"）
      - 边界排除（如"本轮是否应明确排除性能调优项？"）
      - 场景类别缺口（如"未检测到恢复流程——回滚/部分失败路径是否在范围内？"）

   问题格式规则：
   - 如果展示选项，生成紧凑的表格，列为：选项 | 候选项 | 重要性
   - 最多 A-E 选项；如果自由回答更清晰则省略表格
   - 不要要求用户重述已说过的内容
   - 避免推测性分类（不要臆造）。如果不确定，明确询问："确认 X 是否在范围内。"

   无法交互时的默认值：
   - 深度：标准
   - 受众：如果与代码相关则为审查者（PR）；否则为作者
   - 关注点：前 2 个相关性聚类

   输出问题（标记 Q1/Q2/Q3）。回答后：如果 ≥2 个场景类别（替代/异常/恢复/非功能领域）仍不明确，你可以再追问最多两个针对性问题（Q4/Q5），每个附一行理由（如"未解决的恢复路径风险"）。总计不超过五个问题。如果用户明确拒绝更多问题则跳过升级。

3. **理解用户请求**：结合 `$ARGUMENTS` + 澄清答案：
   - 推导检查清单主题（如安全、审查、部署、UX）
   - 整合用户提到的明确必须包含的项目
   - 将关注点选择映射到分类框架
   - 从 spec/plan/tasks 推断缺失的上下文（不要臆造）

4. **加载功能上下文**：从 FEATURE_DIR 读取：
   - spec.md：功能需求和范围
   - plan.md（如存在）：技术细节、依赖
   - tasks.md（如存在）：实现任务

   **上下文加载策略**：
   - 仅加载与活跃关注领域相关的必要部分（避免全文件转储）
   - 优先将长章节总结为简洁的场景/需求要点
   - 使用渐进式披露：仅在检测到缺口时添加后续检索
   - 如果源文档很大，生成中间摘要项而非嵌入原始文本

5. **生成检查清单**——创建"需求的单元测试"：
   - 如果 `FEATURE_DIR/checklists/` 目录不存在则创建
   - 生成唯一的检查清单文件名：
     - 使用基于领域的简短描述性名称（如 `ux.md`、`api.md`、`security.md`）
     - 格式：`[domain].md`
     - 如果文件已存在，追加到现有文件
   - 从 CHK001 开始按顺序编号
   - 每次 `/speckit.checklist` 运行创建一个新文件（不覆盖已有检查清单）

   **核心原则——测试需求，而非实现**：
   每个检查项必须评估需求本身的：
   - **完整性**：所有必要的需求是否都存在？
   - **清晰度**：需求是否无歧义且具体？
   - **一致性**：需求之间是否一致？
   - **可衡量性**：需求是否可以客观验证？
   - **覆盖度**：所有场景/边界情况是否已涵盖？

   **分类结构**——按需求质量维度分组：
   - **需求完整性**（所有必要的需求是否已记录？）
   - **需求清晰度**（需求是否具体且无歧义？）
   - **需求一致性**（需求之间是否无冲突？）
   - **验收标准质量**（成功标准是否可衡量？）
   - **场景覆盖**（所有流程/情况是否已涵盖？）
   - **边界情况覆盖**（边界条件是否已定义？）
   - **非功能需求**（性能、安全、无障碍等——是否已指定？）
   - **依赖与假设**（是否已记录并验证？）
   - **歧义与冲突**（哪些需要澄清？）

   **如何编写检查项——"需求的单元测试"**：

   ❌ **错误**（测试实现）：
   - "验证着陆页显示 3 个剧集卡片"
   - "测试桌面端悬停状态是否有效"
   - "确认点击 logo 导航到首页"

   ✅ **正确**（测试需求质量）：
   - "是否明确指定了精选剧集的确切数量和布局？" [Completeness]
   - "'突出显示'是否用具体的尺寸/定位进行了量化？" [Clarity]
   - "所有交互元素的悬停状态需求是否一致？" [Consistency]
   - "是否为所有交互式 UI 定义了键盘导航需求？" [Coverage]
   - "是否指定了 logo 图片加载失败时的降级行为？" [Edge Cases]
   - "是否为异步剧集数据定义了加载状态？" [Completeness]
   - "规约是否定义了竞争 UI 元素的视觉层次？" [Clarity]

   **检查项结构**：
   每个项目应遵循以下模式：
   - 以问题形式询问需求质量
   - 关注规约/计划中已写（或未写）的内容
   - 在方括号中包含质量维度 [Completeness/Clarity/Consistency/等]
   - 检查现有需求时引用规约章节 `[Spec §X.Y]`
   - 检查缺失需求时使用 `[Gap]` 标记

   **按质量维度的示例**：

   完整性：
   - "是否为所有 API 失败模式定义了错误处理需求？ [Gap]"
   - "是否为所有交互元素指定了无障碍需求？ [Completeness]"
   - "是否为响应式布局定义了移动端断点需求？ [Gap]"

   清晰度：
   - "'快速加载'是否用具体的时间阈值进行了量化？ [Clarity, Spec §NFR-2]"
   - "'相关剧集'的选择标准是否已明确定义？ [Clarity, Spec §FR-5]"
   - "'突出'是否用可衡量的视觉属性进行了定义？ [Ambiguity, Spec §FR-4]"

   一致性：
   - "所有页面的导航需求是否一致？ [Consistency, Spec §FR-10]"
   - "着陆页和详情页之间的卡片组件需求是否一致？ [Consistency]"

   覆盖度：
   - "是否为零状态场景（无剧集）定义了需求？ [Coverage, Edge Case]"
   - "是否涵盖了并发用户交互场景？ [Coverage, Gap]"
   - "是否为部分数据加载失败指定了需求？ [Coverage, Exception Flow]"

   可衡量性：
   - "视觉层次需求是否可衡量/可测试？ [Acceptance Criteria, Spec §FR-1]"
   - "'平衡的视觉权重'是否可以客观验证？ [Measurability, Spec §FR-2]"

   **场景分类与覆盖**（需求质量关注点）：
   - 检查以下场景是否存在需求：主要、替代、异常/错误、恢复、非功能场景
   - 对每个场景类别，询问："[场景类型] 需求是否完整、清晰且一致？"
   - 如果场景类别缺失："[场景类型] 需求是有意排除还是缺失？ [Gap]"
   - 当涉及状态变更时包含弹性/回滚："是否为迁移失败定义了回滚需求？ [Gap]"

   **可追溯性要求**：
   - 最低要求：≥80% 的项目必须包含至少一个可追溯性引用
   - 每个项目应引用：规约章节 `[Spec §X.Y]`，或使用标记：`[Gap]`、`[Ambiguity]`、`[Conflict]`、`[Assumption]`
   - 如果不存在 ID 体系："是否建立了需求和验收标准的 ID 方案？ [Traceability]"

   **发现并解决问题**（需求质量问题）：
   询问关于需求本身的问题：
   - 歧义："术语'快速'是否用具体指标进行了量化？ [Ambiguity, Spec §NFR-1]"
   - 冲突："§FR-10 和 §FR-10a 之间的导航需求是否冲突？ [Conflict]"
   - 假设："'始终可用的播客 API'假设是否已验证？ [Assumption]"
   - 依赖："外部播客 API 需求是否已记录？ [Dependency, Gap]"
   - 缺失定义："'视觉层次'是否用可衡量的标准进行了定义？ [Gap]"

   **内容整合**：
   - 软上限：如果原始候选项 > 40，按风险/影响优先排序
   - 合并检查相同需求方面的近似重复项
   - 如果 > 5 个低影响边界情况，创建一个项目："需求中是否涵盖了边界情况 X、Y、Z？ [Coverage]"

   **🚫 绝对禁止**——这些会使其成为实现测试而非需求测试：
   - ❌ 任何以"验证"、"测试"、"确认"、"检查" + 实现行为开头的项目
   - ❌ 引用代码执行、用户操作、系统行为
   - ❌ "正确显示"、"正常工作"、"按预期运行"
   - ❌ "点击"、"导航"、"渲染"、"加载"、"执行"
   - ❌ 测试用例、测试计划、QA 流程
   - ❌ 实现细节（框架、API、算法）

   **✅ 必需模式**——这些测试需求质量：
   - ✅ "是否为 [场景] 定义/指定/记录了 [需求类型]？"
   - ✅ "[模糊术语] 是否用具体标准进行了量化/澄清？"
   - ✅ "[章节 A] 和 [章节 B] 之间的需求是否一致？"
   - ✅ "[需求] 是否可以客观衡量/验证？"
   - ✅ "需求中是否涵盖了 [边界情况/场景]？"
   - ✅ "规约是否定义了 [缺失方面]？"

6. **结构参考**：按照 `.specify/templates/checklist-template.md` 中的规范模板生成检查清单，包括标题、元信息章节、分类标题和 ID 格式。如果模板不可用，使用：H1 标题、用途/创建日期元信息行、`##` 分类章节包含 `- [ ] CHK### <需求检查项>` 行，ID 从 CHK001 开始全局递增。

7. **报告**：输出创建的检查清单完整路径、项目数量，并提醒用户每次运行创建新文件。总结：
   - 选择的关注领域
   - 深度级别
   - 受众/时机
   - 纳入的用户明确指定的必须包含项

**重要**：每次 `/speckit.checklist` 命令调用使用简短描述性名称创建检查清单文件（除非文件已存在）。这允许：

- 不同类型的多个检查清单（如 `ux.md`、`test.md`、`security.md`）
- 简单、易记的文件名，表明检查清单用途
- 在 `checklists/` 文件夹中易于识别和导航

为避免混乱，使用描述性类型，完成后清理过时的检查清单。

## 检查清单类型示例与样本项

**UX 需求质量：** `ux.md`

样本项（测试需求，而非实现）：

- "是否用可衡量的标准定义了视觉层次需求？ [Clarity, Spec §FR-1]"
- "是否明确指定了 UI 元素的数量和定位？ [Completeness, Spec §FR-1]"
- "交互状态需求（悬停、聚焦、激活）是否一致定义？ [Consistency]"
- "是否为所有交互元素指定了无障碍需求？ [Coverage, Gap]"
- "是否定义了图片加载失败时的降级行为？ [Edge Case, Gap]"
- "'突出显示'是否可以客观衡量？ [Measurability, Spec §FR-4]"

**API 需求质量：** `api.md`

样本项：

- "是否为所有失败场景指定了错误响应格式？ [Completeness]"
- "限流需求是否用具体阈值进行了量化？ [Clarity]"
- "所有端点的认证需求是否一致？ [Consistency]"
- "是否为外部依赖定义了重试/超时需求？ [Coverage, Gap]"
- "需求中是否记录了版本策略？ [Gap]"

**性能需求质量：** `performance.md`

样本项：

- "性能需求是否用具体指标进行了量化？ [Clarity]"
- "是否为所有关键用户旅程定义了性能目标？ [Coverage]"
- "是否指定了不同负载条件下的性能需求？ [Completeness]"
- "性能需求是否可以客观衡量？ [Measurability]"
- "是否为高负载场景定义了降级需求？ [Edge Case, Gap]"

**安全需求质量：** `security.md`

样本项：

- "是否为所有受保护资源指定了认证需求？ [Coverage]"
- "是否为敏感信息定义了数据保护需求？ [Completeness]"
- "威胁模型是否已记录且需求与之对齐？ [Traceability]"
- "安全需求是否与合规义务一致？ [Consistency]"
- "是否定义了安全故障/违规响应需求？ [Gap, Exception Flow]"

## 反面示例：不应该做什么

**❌ 错误——这些测试实现，而非需求：**

```markdown
- [ ] CHK001 - 验证着陆页显示 3 个剧集卡片 [Spec §FR-001]
- [ ] CHK002 - 测试桌面端悬停状态是否正确 [Spec §FR-003]
- [ ] CHK003 - 确认点击 logo 导航到首页 [Spec §FR-010]
- [ ] CHK004 - 检查相关剧集章节显示 3-5 个项目 [Spec §FR-005]
```

**✅ 正确——这些测试需求质量：**

```markdown
- [ ] CHK001 - 是否明确指定了精选剧集的数量和布局？ [Completeness, Spec §FR-001]
- [ ] CHK002 - 所有交互元素的悬停状态需求是否一致定义？ [Consistency, Spec §FR-003]
- [ ] CHK003 - 所有可点击品牌元素的导航需求是否清晰？ [Clarity, Spec §FR-010]
- [ ] CHK004 - 相关剧集的选择标准是否已记录？ [Gap, Spec §FR-005]
- [ ] CHK005 - 是否为异步剧集数据定义了加载状态需求？ [Gap]
- [ ] CHK006 - "视觉层次"需求是否可以客观衡量？ [Measurability, Spec §FR-001]
```

**关键区别：**

- 错误：测试系统是否正确运行
- 正确：测试需求是否正确编写
- 错误：行为验证
- 正确：需求质量验证
- 错误："它是否做了 X？"
- 正确："X 是否被清晰地指定了？"
